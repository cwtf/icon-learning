# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

Marketing/catalog website for Icon Learning & Development Sdn Bhd, a Malaysian corporate training and ISO consultancy provider. Astro static site, deployed to `www.iconlearning.com.my`. Redesign of a legacy site (see `dns-migration.md`, `docs/launch/redirect-plan.md`).

Two files carry the running project context and must be read before non-trivial work:

- **`PROJECT_BRAIN.md`** — running state log (status, what's done, in-flight, decisions log, open questions). New sessions should read this first. Update it when a PR-sized change lands.
- **`DESIGN.md`** — source-of-truth spec (IA, visual system, component contracts, content model, SEO recipe, acceptance criteria). Don't duplicate its content into `PROJECT_BRAIN.md`; point at the section number instead.

## Commands

```bash
npm run dev                    # astro dev server
npm run build                  # astro check + astro build (prebuild runs sitemap generation)
npm run preview                # preview production build
npm run typecheck              # astro check only

npm run courses:inventory      # scan course/documents-markdown/, regenerate _curation/inventory.json
npm run courses:review         # validate all published course JSON against schema rules; exit 1 on hard errors
npm run courses:draft          # course draft generation helper
npm run courses:checklist      # course markdown checklist helper
npm run courses:seo-polish     # course SEO polish helper
npm run documents:markdown     # pandoc conversion of source documents to markdown
npm run qa:launch              # post-build QA: run AFTER `npm run build` — checks dist/ meta tags, JSON-LD, H1s, links, image alt/dimensions, crawl files, script gzip budget; writes dist/launch-qa-report.json
npm run sitemaps:generate      # regenerate sitemap-pages.xml / sitemap-courses.xml / sitemap index (also runs as prebuild)
```

There is no test runner in this repo; correctness is enforced by `astro check`, `courses:review`, and `qa:launch`.

## Architecture

**Stack**: Astro (static output) + Tailwind + hand-rolled `tokens.css` CSS variables. No client framework — interactivity is isolated to small vanilla-TS islands loaded via `<script>` tags (`*.client.ts` files next to their `.astro` component). Client JS budget is enforced by `qa:launch` (~50 KB gzip for the whole homepage).

Path aliases (`tsconfig.json`): `@components/*`, `@content/*`, `@layouts/*`, `@styles/*`.

### Content model

- **`src/content/courses/*.json`** — one file per publishable course, the catalog's source of truth. Shape defined by `src/content/courses/schema.ts` (`Course` type: category slug, duration, `hrdClaimable`, language, delivery formats, `whyThisMatters`, `builtFor`, `walkAwayWith`, module titles, methodology mix, `relatedSlugs`). `src/content/courses/index.ts` globs `./*.json` (non-recursive) to build the catalog — files under `_curation/` are intentionally excluded from that glob and never rendered.
- **`src/content/courses/_curation/`** — pipeline scratch space, not publishable content: `inventory.json` (regenerated by `courses:inventory` from raw source docs) and `review-report.json` (regenerated by `courses:review`).
- Raw source course PDFs/docs live under the gitignored `course/` directory (not in the repo) — never expected to exist in a fresh checkout.
- **Course rewrite pipeline**: raw docs → `documents:markdown` (pandoc) → `courses:inventory` (groups/dedupes into candidate courses) → human-authored `*.json` per course following `schema.ts` → `courses:review` (validates). Always run `courses:review` after adding/editing a course JSON file.
- `hrdClaimable` must default to `false` unless HRD Corp claimable status is confirmed from the source PDF or an authoritative list — do not infer it.
- Course detail pages are **summary-only by design**: module titles, not topic-by-topic outlines from source PDFs. This is a deliberate content-model constraint (see `DESIGN.md` §6.4, §9.3), not an oversight — don't "complete" course pages with full outline dumps.
- **`src/content/programs.ts`** — the 10 program category definitions (`courseCategorySlugs` in `schema.ts` is the canonical slug list; keep both in sync). Also carries category-page copy (overview, focus areas, audience, related categories).
- Other `src/content/*.ts` files (`site.ts`, `home.ts`, `about.ts`, `clients.ts`, `contact.ts`, `testimonials.ts`, `navigation.ts`, `accessibility.ts`, `publicTraining.ts`) hold page copy as typed data, kept separate from `.astro` templates. `site.ts` is the single source for NAP (name/address/phone) — keep it in sync with footer, `/contact`, and `LocalBusiness` JSON-LD (drift hurts local SEO).

### Routing

- `/` , `/about-us`, `/clients`, `/contact`, `/accessibility-statement` — static pages in `src/pages/`.
- `/programs` — full catalog with client-side search/filter (`ProgramsCatalog.client.ts`); grouped/flat views, HRD-claimable toggle, URL query state. Progressive enhancement — readable as static HTML without JS.
- `/programs/[category]` — static, `getStaticPaths()` from `programCategories`, no JS.
- `/programs/[category]/[course]` — static, one page per course JSON.
- `/register/[course]` — registration flow; submits to a Google Apps Script endpoint (`PUBLIC_GOOGLE_SHEET_REGISTRATION_ENDPOINT` in `.env`, see `scripts/google-sheet-registration-web-app.gs` for the receiving script).

### Components (`src/components/`)

- `layout/` — `Nav`, `Footer` (site chrome, used by `BaseLayout`).
- `primitives/` — `Button`, `Card`, `SectionHeading`, `TabBar`, `LogoWall`, `BrandLogo`. Reusable, no page-specific logic. Variant contracts are documented in `DESIGN.md` §5.1 — check there before adding a new variant.
- `sections/` — homepage sections (`Hero`, `BentoMasonry`, `CoursesTabbed`, `ApproachSection`/`FeatureStripe`, `ServicesBento`, `Testimonials`, `CtaCloser`, `HomeClients`, `FeaturedPublicTraining`). Each interactive section pairs a `.astro` (markup, SSR'd, no-JS-safe fallback) with a `.client.ts` island of the same name.
- `programs/`, `contact/`, `register/` — feature-scoped components for those routes.
- `ResultsGrid.astro` is a deliberate no-op placeholder — the homepage outcomes segment was removed because verified stats weren't available; do not resurrect it without checking `PROJECT_BRAIN.md`'s decisions log first.

### Conventions worth knowing before editing

- **Motion is progressive enhancement**: `src/scripts/global-motion.ts` adds `data-motion-ready` to `<html>`; reveal/stagger CSS only activates after that. No-JS users see fully visible content. The script no-ops entirely under `prefers-reduced-motion: reduce`.
- **No fabricated content**: don't add trainee/client counts, testimonial quotes, or outcome numbers that aren't verified. This project has an explicit history (see `PROJECT_BRAIN.md` decisions log) of removing sections built on unverified claims.
- **SEO is templated, not artisan**: title/meta description patterns per page type, JSON-LD block requirements (global `Organization`/`WebSite` via `BaseLayout`; page-specific `LocalBusiness`, `Course`, `BreadcrumbList`, `ItemList`, `FAQPage`) are specified in `DESIGN.md` §9. Follow the templates; deviations should be deliberate.
- **HRD Corp claimable** is the primary commercial/search differentiator — treat it as a first-class field wherever courses are surfaced (filters, meta descriptions, structured data), not a footnote.
- Analytics (GA4 + Microsoft Clarity) and global structured data are wired directly into `src/layouts/BaseLayout.astro` — all pages should route through this layout.
- Light theme by default, dark theme follows `prefers-color-scheme`, with a manual toggle in `Nav.astro` that sets `data-theme` on `<html>` (persisted to `localStorage.theme` only when overriding the system). Both palettes live in `src/styles/tokens.css` (the two dark blocks must stay identical). All color must flow through tokens or `color-mix()` over tokens — never raw hex/rgb in components. No glassmorphism, sparse purple accent only; see `DESIGN.md` §4 before introducing new colors.
